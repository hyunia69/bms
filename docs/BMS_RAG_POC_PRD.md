# BMS 공조 시스템 RAG 기반 지능형 모니터링 POC

## 문서 정보

- 작성일: 2026-05-20
- 상태: 초안 (Claude Code 브레인스토밍 전)
- 목적: POC 구현을 위한 방향 정의

---

## 1. 배경 및 목적

### 배경

공장 빌딩의 공조(HVAC) 시스템은 AHU, FCU, 냉동기, 냉각탑, 순환펌프 등 다수의 기기로 구성되어 있다. 현재 이 기기들의 상태 로그는 지속적으로 생성되지만, 단순한 수치 모니터링에 머물러 있어 다음과 같은 문제가 있다.

- 에러 발생 시 코드만 표시되고 원인/조치 정보가 없음
- 복합적인 에러 패턴의 상관관계를 파악하기 어려움
- 현장 운전원이 매뉴얼을 직접 찾아봐야 하는 번거로움
- 야간/비상 상황에서 의사결정 지원이 부족함

### 목적

RAG(Retrieval-Augmented Generation) 기술을 활용하여 기기 로그 데이터와 HVAC 기술 문서를 연결하고, 이상 감지부터 원인 분석, 조치 가이드까지 자동화된 지능형 모니터링 시스템의 가능성을 검증한다.

## 2. POC 범위

### 핵심 기능 (Must Have)

**기능 1. 로그 생성 시뮬레이터**

실제 공장 공조 시스템에서 발생하는 것과 유사한 로그를 자동 생성한다. POC 검증을 위한 것으로, 실제 BMS 시스템 연동 전 단계에서 활용한다.

- 공조 기기별 상태 데이터 생성 (온도, 습도, 압력, 전류, 풍량 등)
- 정상/경고/오류/위험 상태를 현실적인 비율로 생성
- 시간대별 외기 온도 패턴 반영
- 특정 에러 상황을 강제로 주입할 수 있는 기능

**기능 2. 로그 수집 및 저장**

생성된 로그를 구조화된 형태로 저장하고 조회할 수 있어야 한다.

- 기기별, 시간별 로그 적재
- 에러코드, 상태값, 타임스탬프 포함
- 분석을 위한 집계 쿼리 지원

**기능 3. 주기적 로그 분석**

일정 주기로 누적된 로그를 분석하여 시스템 상태를 요약한다.

- 기기별 정상/경고/오류 현황 집계
- 이상 패턴 감지 (단순 임계값 초과 외 트렌드 분석 포함)
- 분석 결과 저장 및 이력 관리

**기능 4. RAG 기반 문서 검색 및 분석**

이상 감지 시 관련 기술 문서를 검색하여 원인과 조치 방법을 제공한다.

- HVAC 에러코드, 점검 절차, 운전 매뉴얼 등을 벡터 임베딩으로 인덱싱
- 감지된 에러/이상 패턴과 관련 문서를 의미 기반으로 매칭
- 검색된 문서를 컨텍스트로 활용하여 AI 분석 생성
- 단순 규칙 기반 알림을 넘어 원인 추론 및 조치 권고 제공

**기능 5. Slack 알림**

이상 감지 시 담당자에게 즉시 알린다.

- 오류 수준(경고/오류/위험)에 따른 알림
- RAG 분석 결과 포함 (원인 분석, 즉각 조치사항)
- 기기 위치, 발생 시각, 에러코드 포함

**기능 6. Slack 자연어 질의응답**

운전원이 Slack에서 자연어로 질문하면 RAG 기반으로 답변한다.

- "@봇 질문" 형식으로 언제든 질의 가능
- 질문을 임베딩으로 변환하여 관련 기술 문서 검색
- 검색된 문서 기반으로 AI가 맥락에 맞는 답변 생성
- 현재 기기 상태 데이터와 결합하여 상황에 맞는 가이드 제공
- Slack 스레드에 답변 (대화 맥락 유지)

### 선택 기능 (Nice to Have)

- 일간/주간 분석 보고서 자동 생성 및 이메일 발송
- Notion에 분석 이력 동기화
- 임베딩 공간 시각화 (유사도 분포 확인)
- 에너지 효율 이상 감지 (에러 없는 전력 급증)

### 범위 외

- 실제 BMS 시스템 직접 연동
- 실시간 스트리밍 처리 (배치 처리로 대체)
- 모바일 앱
- 사용자 인증/권한 관리

## 3. 핵심 시나리오

### 시나리오 A — 자동 감지 및 알림

```
공조 기기에서 이상 로그 발생
  → 주기적 분석 실행
  → 이상 패턴 감지
  → 관련 기술 문서 RAG 검색
  → AI가 원인 분석 및 조치 권고 생성
  → Slack 채널에 분석 결과 포함 알림 전송
```

### 시나리오 B — 운전원 자연어 질의

```
운전원: "@BMS봇 CHILLER-01 지금 기계실 내려가야 해요?"
  → 질문 임베딩 변환
  → 관련 기술 문서 유사도 검색
  → 현재 기기 상태와 결합
  → AI가 상황에 맞는 답변 생성
  → Slack 스레드에 답변
```

### 시나리오 C — 복합 에러 패턴 분석

```
여러 기기에서 동시다발 경고 발생
  → 단순 에러 감지가 아닌 상관관계 분석
  → RAG로 복합 패턴 관련 문서 검색
  → 공통 근본 원인 추론
  → 우선 조치 순서 권고
```

## 4. RAG 구성 요소

### 베이스 문서 (인덱싱 대상)

POC에서 사용할 기술 문서의 종류와 내용이다. 실제 제조사 매뉴얼이 있으면 활용하고, 없으면 생성한다.

**에러코드 레퍼런스**
- 각 에러코드별 발생 조건, 가능한 원인, 즉각 조치, 관련 코드
- 경고/오류/위험 수준별 대응 기준

**점검 및 수리 절차서**
- 필터 교체, 냉매 점검, 압축기 점검 등 주요 작업 절차
- 긴급 정지 및 비상 대응 절차
- 계절별 운전 전환 절차

**기기 운전 매뉴얼**
- AHU, FCU, 냉동기, 냉각탑별 정상 운전 범위
- 주요 파라미터 기준값 및 허용 범위
- 정기점검 항목 및 주기

**에너지 관리 가이드**
- 정상 에너지 소비 패턴
- 효율 저하 판단 기준
- 에너지 절약 운전 방법

### RAG 핵심 동작

```
[사전 준비 — 인덱싱]
문서 수집/생성
  → 의미 단위로 청크 분할
  → 임베딩 모델로 벡터 변환
  → 벡터 DB에 저장

[실시간 — 검색 및 생성]
질문 또는 에러 정보
  → 임베딩 변환
  → 벡터 유사도 검색 (의미 기반 매칭)
  → 관련 문서 Top-K 반환
  → LLM에 문서 + 질문 주입
  → 문서 기반 답변 생성
```

## 5. 데이터 구조 (개념)

### 원시 로그

기기에서 주기적으로 수집되는 데이터

- 수집 시각
- 기기 식별자 및 종류
- 설치 위치 (구역)
- 상태 (정상/경고/오류/위험)
- 에러코드
- 온도 (급기/환기/외기)
- 습도, 압력, 풍량, 전류, 전력
- 누적 가동시간
- 상세 메시지

### 분석 결과

주기적 분석 실행 후 저장하는 요약 데이터

- 분석 실행 시각 및 대상 기간
- 기기별 상태 요약
- 감지된 이상 목록
- AI 분석 텍스트
- 조치 권고사항
- 알림 발송 여부 및 수준

### RAG 문서

벡터 임베딩과 함께 저장되는 기술 문서

- 제목
- 내용
- 문서 종류 (에러코드/절차서/매뉴얼)
- 관련 기기 종류
- 벡터 임베딩
- 메타데이터

## 6. 주요 제약 및 고려사항

### 기술적 제약

- POC 수준으로 구현 복잡도를 최소화
- 셀프호스팅 가능한 구성 우선 고려
- 기존에 보유한 계정/서비스 활용 (Supabase, Slack 등)

### 비기능 요구사항

- 알림 응답 시간: 에러 감지 후 1분 이내 Slack 알림
- Slack 질의 응답 시간: 질문 후 30초 이내 답변
- RAG 문서 검색 정확도: 관련 문서가 Top-3 안에 포함

### 확장 가능성

POC 이후 실제 시스템으로 확장 시 고려할 것들

- 실제 BMS 시스템 API 연동
- 실시간 스트리밍 처리
- 다중 건물/공장 지원
- 모바일 알림 연동
- 대화형 운전 제어 (알림 → 제어 명령)

## 7. 성공 기준

POC가 성공했다고 판단하는 기준이다.

**필수 검증 항목**

- 로그 시뮬레이터가 현실적인 HVAC 로그를 생성하는가
- 에러 발생 시 Slack에 원인 분석 포함 알림이 전송되는가
- Slack에서 자연어 질문 시 관련 문서 기반 답변이 나오는가
- 키워드가 없어도 의미 기반으로 관련 문서가 검색되는가

**검증 시나리오 예시**

- "냉매가 새는 것 같아요" → E-3002 관련 문서 검색
- "필터 언제 갈아요?" → 필터 교체 절차 문서 검색
- "새벽에 온도가 갑자기 올라갔어요" → 급기온도 에러 + 계절 운전 문서 검색

## 8. 브레인스토밍 어젠다 (Claude Code 시작 시)

Claude Code에서 구현 전 논의할 사항들이다.

**아키텍처 결정**
- 워크플로우 자동화 도구 선택 및 구성 방식
- 벡터 DB 및 임베딩 모델 선택
- LLM 선택 (Claude / GPT-4o 등)
- 로그 저장소 선택

**RAG 설계**
- 청크 분할 전략 (에러코드 단위 vs 절 단위)
- 임베딩 차원 및 모델 선택
- 유사도 임계값 설정
- RAG 문서 초기 세트 구성

**Slack 연동 설계**
- 알림 채널 구성 (전체/기기별/수준별)
- 질의응답 봇 설계
- 스레드 vs 새 메시지 전략

**시뮬레이터 설계**
- 기기 구성 (종류, 수량, 구역)
- 에러 발생 빈도 및 패턴
- 시간대별 부하 패턴

## 9. 참고 컨텍스트

이 PRD 작성 전 논의에서 결정된 방향들이다. Claude Code 브레인스토밍 시 참고하되 변경 가능하다.

- 워크플로우 자동화로 Activepieces를 검토했으나 확정은 아님
- 벡터 DB로 Supabase pgvector를 검토했으나 확정은 아님
- Slack 무료 플랜에서도 Incoming Webhook 및 Event Subscriptions 사용 가능
- RAG 문서는 Claude로 생성하거나 실제 제조사 문서 활용
- 로그 생성 주기(1분), 분석 주기(1시간)는 POC에서 조정 가능
